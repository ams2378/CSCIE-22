  1. Problem 1 requires us to draw a memory diagram. What file format should we submit this diagram in?

    Generally speaking, for the Part I problems you should submit your work in a plain-text file. Because we are asking you to draw a diagram, you may submit either a plain-text file or a PDF file.

    If you are using a PDF file, you can create the diagram using the tool of your choice and embed it in the PDF. If you are using a plain-text file, you can create the diagram using “ASCII art”. Here is an example of a perfectly good ASCII memory diagram:

        Stack:           Heap:
        ----         -----------------
    arr | -|-------->| 1 | 2 | 3 | 4 |
        ----         -----------------

  2. When I run one of the recursive methods that I’ve written for problem 7, I get a NullPointerException. Do you have any idea of why that might be happening?

    A NullPointerException occurs when you try to call a method using a reference variable with a value of null. For example, if you have a variable named str with a value of null and you call str.length(), you will get this exception because there is no object on which the length() method can be invoked. You may need to check first if the value of a reference variable is null before attempting to use it to call a method.

  3. In one of my recursive methods for problem 7, I’m getting a StringIndexOutOfBoundsException. What am I doing wrong?

    Don’t forget that the characters in a string have index values that go from 0 to length - 1. The exception means that your code is using an index from outside that range.

  4. In one of my recursive methods for problem 7, I’m using the substring method to create a substring that has everything but the first character from the original string. I’m using the expression str.substring(1, str.length() - 1), but I seem to be losing characters at the end of the string as well.

    Don’t forget that the second parameter to the substring method is non-inclusive. So if you want the last character to be included in the substring, your should use the expression str.substring(1, str.length()) – without the “- 1”. Alternatively, you could use the version of substring that only takes a start index: str.substring(1).

  5. I’m having trouble figuring out how to structure the recursive case of one of the recursive methods for problem 7. Do you have any hints on how to do this?

    One thing that can help is to consider what should happen for one or more concrete cases – as we did in the example problem that we covered in lecture.

    For example, let’s say that you needed to write a recursive removeVowels method. As with many recursive methods that operate on strings, this method should make recursive calls on ever smaller substrings. For example, let’s say that we wanted to do removeVowels("movies")

    This would lead to the following sequence of method calls:

    removeVowels("movies")
    removeVowels("ovies")
        removeVowels("vies")
            removeVowels("ies")
                removeVowels("es")
                    removeVowels("s")
                        removeVowels("")

    (The last call might not be needed. It depends on which base cases you choose to include.)

    Then, once you have the sequence of method calls, you should think about what each of these separate method calls should return, treating them as if they were independent of each other. For example, what should removeVowels("ies") return – i.e., what string will result if the vowels in "ies" were removed? Based on these return values, you should be able to figure out how a given invocation of the method should use the return value from the recursive call to form its own return value.

  6. One of my recursive methods for problem 7 is not working correctly. Do you have any suggestions?

    Try tracing through some concrete examples of cases in which the your method is not returning the correct value. You might want to try adding some temporary printlns to see if that helps you to diagnose the problem. In particular, you could print what the method will return before it actually returns it. That will allow you to see when the wrong value is being returned.

    In addition, if your method returns a value, make sure that you aren’t throwing away the return value of the recursive call. For example, consider this incorrect version of a recursive method that determines if a string is a palindrome – i.e., if it is a word like “radar” that reads the same in either direction: 

    public static boolean isPalindrome(String str) {  
    if (str == null || str.equals("")) {  
        return true;  
    }

    // recursive call -- the return value is thrown away
    isPalindrome(str.substring(1, str.length() - 1));

    char first = str.charAt(0);
    char last = str.charAt(str.length() - 1);
    if (first != last) {
        return false;
    } else {
        return true;
    }
}

    This version of the method makes the correct recursive call – looking at the substring consisting of everything except the first and last characters – but it throws away the value that is returned.

    To avoid losing the return value of the recursive call, we can do one of two things:

    1. assign it to a variable, and then do something with that variable
    2. make the recursive call part of a return statement, if doing so makes sense. It may not always make sense – especially if the value that the current method call should return depends on the value that was returned by the recursive call.

  7. I’m having trouble with the Lagrange problem. Can you give us any hints on how to get started?

    Carefully read over the example that we’ve given you in the problem itself (the one in which we break up 140). It should give you some insight about how to structure your method – including what base case(s) you will need, and what your recursive call should look like.

    Don’t forget that a given call of the recursive method is responsible for determining the value of one of the variables that you are working with (in this case, one of the terms of the sum), and that the method should use a loop to try all possible values for that variable.

  8. I am struggling to apply the backtracking template from the lecture notes to the Lagrange problem. I’m confused about what n and val in the template should mean in the context of this problem. Also, is an isValid() method needed here?

    The template is useful, but you shouldn’t get too bogged down in the details of it. Focus first on the basics of recursive backtracking and on the example given in the problem itself. You should see that the template fits pretty well – although your solution may end up differing from the exact structure shown in the template.

    As we said in lecture, recursive backtracking is a method for assigning values to a set of variables in light of a set of constraints. In the template, n tells you the variable that you’re currently trying to assign a value to, and val iterates over the set of possible values for that variable. (Note that n is not the variable itself - it is simply a parameter that allows you to determine the variable. For example, in the Queens problem, n was the number of row in which we were trying to place a queen, but we stored the actual values—i.e., the locations of the queens—in other data structures.)

    The isValid() method in our template simply checks if val is a valid value for the nth variable (in other words, does it fit the constraints?). Depending on how you structure your solution, you may not need to actually write a method for this.

    Another detail of the template that can vary is the way that the parameters to the method are modified in the recursive call. In the template, n is incremented by 1 to indicate that the recursive call will focus on the next variable. However, if you take the approach that we have recommended in the problem (and illustrated in the example), you will need to modify the parameters in some other way when you make the recursive call. The important thing is that the new parameters should reflect the subproblem that you’re trying to solve by making the recursive call, and the parameters should eventually reach one of the values that constitute a base case of the recursion.

  9. I’m still struggling on the Lagrange problem. Do you have any additional suggestions?

    Here are four things that may help:

    • If you look the example that we’ve given you in the problem set, you’ll notice that a given call of findSumRB() is responsible for determining the value of just one of the terms of the sum. For example, findSumRB(140,4) is responsible for the first term in the sum (terms[0]), and findSumRB(19,3) is responsible for the second term in the sum (terms[1]). There is an easy piece of arithmetic that you can use to determine which term a given call of the method is responsible for, based on its parameters.

    • In attempting to determine the value of a given term, you need to try the perfect squares that could work for that term, and we’ve recommended that you start with the largest perfect square that would work. For example, say you are looking to break up 29 (i.e., num is 29). The sequence of squares that you would try is 25, 16, 9, 4, 1; i.e., those are the numbers you should loop over in that method frame. Because it may not be immediately clear how to set up a loop such that that sequence of squares is tried, you may want to consider looping over the roots of the squares instead of the squares themselves. In that case, the loop would iterate over: 5, 4, 3, 2, 1.

    • You may find it helpful to have more than one base-case test at the start of findSumRB: one to handle the case in which you have found a solution, and one to handle the case in which you have run out of terms and need to backtrack. (Note that this instance of backtracking is in addition to the backtracking that occurs once you have tried all possible values for a given term.)

    • Although you want the recursive call to have modified parameter values, you need to make sure that you don’t modify the values of the parameters in the current stack frame. For example, in the nQueens example from lecture, we make a recursive call with a parameter of row+1. That gives the recursive call a parameter that is one more than the current call’s parameter, but it does so in a way that doesn’t change the value of row in the current stack frame. If we had instead done row++ or ++row, we would have modified the value of row in the current stack frame, and that would have caused problems when we returned to that frame during backtracking.

  10. I’m not really sure how to approach the final problem (“making it smarter”). Should I be looking out for specific numbers when the recursion reaches the 3rd term? Or is there some sort of more general formula that I can use?

    For the final problem, we’re only looking for a simple enhancement (e.g., a single if statement) that causes the backtracking to happen sooner than it otherwise would in certain impossible situations.

    There are multiple approaches you could take that fit this description. Obviously, your enhancement does not need to detect all impossible situations, but it should ideally be a bit more general than just looking out for specific numbers when you reach the 3rd term.

    Think about how you can use the information you have in a given recursive invocation (the number of terms remaining, the amount of the original number remaining, the perfect square you’re considering placing in the current position, etc.) to detect at least one class of impossible situations...
